# 設計徹底レビュー（Opus サブエージェント実行）

このコマンドが呼ばれたら、**Agent ツール**を使って Opus モデルのサブエージェントに設計レビューを委譲する。メイン会話のモデルは変更しない（エージェントから戻った後、元のモデルで作業を続ける）。

## 実行手順

### ステップ1: Agent ツールでレビューを起動

以下のパラメータで Agent ツールを呼ぶ:

- `subagent_type`: `"general-purpose"`
- `model`: `"opus"`（最新の Opus 4.7 が使われる）
- `description`: `"design.md 徹底レビュー"`
- `prompt`: 下記の「エージェントへの指示」セクションの内容をそのまま渡す

### ステップ2: Agent の戻り値を受け取る

Agent は発見した課題の一覧を返す。メイン会話ではその結果を整形してユーザーに報告する。

### ステップ3: ユーザーの了承を得て tasks.md に書き出す

メイン会話（元のモデル）で以下を実施:
1. 発見課題を「N 回目レビューで発見した追加ギャップ」セクションとして `docs/tasks.md` に追記
2. N は既存の最大 +1、タグは W/X/Y/Z/AA... と継続
3. 書き出し後、各課題を一つずつ design.md で解決

### ステップ4: ループの継続

0 件になったら再度 `/design-review` を実行。**本当に 0 件が出続ける**まで繰り返す。過去に 4 回「設計完了」と誤宣言した履歴あり。安易に判断しない。

## エージェントへの指示（Agent の prompt にそのまま渡す）

以下のテキストを Agent ツールの `prompt` パラメータに渡す:

```
あなたは設計ドキュメントの徹底レビューアです。

## タスク

`docs/design.md`（このリポジトリのメインの設計書）を全文精読し、内部矛盾・未定義項目・実装判断が付かない曖昧な点を洗い出してください。ただし、設計よりも後工程で実施すべきことには踏み込まないこと。

## 前提コンテキスト

- このプロジェクトは社内向け動画バッチ編集ツール（Tauri 2.x + React + FFmpeg）
- 要件定義: `docs/requirements.md`
- 設計書: `docs/design.md`（レビュー対象）
- 既存の課題一覧: `docs/tasks.md`

過去に Q1-Q25 / R1-R23 / S1-S16 / T1-T13 / U1-U8 / V1-V13 の計98件が発見・解決済み。`docs/tasks.md` の「レビューで発見した追加ギャップ」セクションを最初に読み、過去と重複する指摘を避けてください。

## レビュー手順

1. `wc -l docs/design.md` で行数を確認
2. `docs/tasks.md` の既存レビューセクションを読み、過去指摘を把握
3. `docs/design.md` を **全行精読**（飛ばし読み禁止。複数回の Read で OK）
4. 以下のカテゴリで系統的に課題を検出

### 🔴 ブロッカー（実装を止める・出力品質に直接影響）
- 内部矛盾（同じ項目が複数箇所で違う値）
- 行番号参照が古い（`L1234` が別セクションを指す）
- 型定義と実装記述の不整合
- Rust コンパイルエラーになる記述（予約語衝突など）
- 壊れた外部参照
- プラットフォーム固有の落とし穴
- 循環参照・論理的に定義不能なケース

### 🟠 重要（実装判断が付かない）
- 具体的な値が未定
- UI 要素の配置・動作が未定義
- IPC・型のフィールド欠落
- エラー処理のパス未定義

### 🟡 軽微（改善）
- 表記スタイル不整合
- コメント欠落
- サンプルコードの一般形と具体例の齟齬

## 機械的チェック（必ず実行）

- Grep で `L[0-9]{3,4}` → 古い行番号参照を検出
- Grep で 空コードブロック（```json\n```等）を検出
- YAML サンプル / TypeScript 型 / Rust 構造体の3者比較（フィールド名・型・必須/任意）
- ディレクトリ構造図とドキュメント本文で言及されるパスの整合
- すべての IPC コマンドの戻り値型が定義されているか
- すべての Tauri イベントペイロード型が定義されているか
- `Cargo.toml` / `package.json` / `tauri.conf.json` の統合サンプルが存在するか
- すべてのショートカットがメニュー項目と対応するか
- FFmpeg コマンド例のプレースホルダスタイル統一

## 出力フォーマット

以下の形式で Markdown テキストとして返してください（tasks.md に貼り付け可能な形式）:

```
## N 回目レビューで発見した追加ギャップ

> YYYY-MM-DD（N 回目レビュー）で発見。design.md 全文（XXXX行）を精読した結果、ブロッカーX件・重要Y件・軽微Z件が残っている。

### 🔴 ブロッカー

- [ ] **W1** （1行要約） — （詳細と修正方針）
- [ ] **W2** ...

### 🟠 重要

- [ ] **W3** ...

### 🟡 軽微

- [ ] **W4** ...
```

N はユーザーが次回指定（現時点では 10 回目想定）、タグプレフィックスは W から始める（V が既使用）。

## 重要な注意

- **過去と重複する指摘は絶対にしない**。tasks.md の過去セクションを最初に読むこと。ただし繰り返された修正により過去と同等の指摘をせざるを得ない状態になっている場合はその限りではない。
- **課題がなければ「0 件」と明確に報告する**。これは設計完了可能な条件の一つ
- **tasks.md への書き込みは行わない**。発見した課題のテキストだけを返す（メイン会話が書き込みを判断する）
- **design.md の修正も行わない**。レビューと発見に徹する
- 報告は 400 語以内に収める（課題が多ければリスト形式で）
```
